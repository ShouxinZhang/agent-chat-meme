# 方法型技术笔记写作约定

适用于 KF / EKF 这类“既要讲朴素直觉，又要保留严格推导”的 TeX 文档。

## 核心原则

- 先讲问题，再讲公式。不要从符号出发，要从“现在遇到了什么问题、为什么非解决不可”出发。
- 正文必须同时有两层：朴素直觉层 + 严格推导层。二者不是二选一。
- 详细推导可以拆到独立页或子 PDF，但正文不能只留结论，必须保留来龙去脉。
- 符号第一次出现时就解释；如果篇幅长，在汇总页再把符号翻回人话。
- 少用自以为通俗的口语词。优先“朴素但严格”，不要“俏皮但含混”。

## 推荐结构

1. 先用第一性原理说明：当前到底想估计什么，手里有什么信息，还缺什么。
2. 再把整条主线拆成连续小问题。
3. 每个小问题先给一句朴素想法，再给正式推导。
4. 最后给一个带注释的公式汇总和一个能手算的例子。

## 具体写法

### 1. 先有大纲，再落公式

- 不要“状态空间模型写完 -> 突然跳到 Kalman 增益”。
- 要先交代：预测解决什么，更新解决什么，当前这一步为什么自然会引出下一个量。

### 2. 符号不能先用后讲

- `K_t`、`S_t` 这种符号，不能突然出现。
- 正确做法是：先说“那个负责把残差翻译成状态修正量的矩阵”，再命名成 `K_t`。
- 对长文档，4.5 这类汇总页要有“符号速查层”。

### 3. 朴素直觉和严格证明要并排出现

- 推荐句式：
  - “先说朴素想法……”
  - “不过这还只是直觉，下面给严格推导。”
- 例如 Kalman 增益这段：
  - 先说“残差里别留有没用干净的信息”
  - 再把目标函数 `J(K)` 写出来，展开，求导，令 0

### 4. 推导优先走最直接的路

- 如果本质是“含参数函数求极小值”，就直接写目标函数、展开、求导、令 0。
- 不要为了“高级感”绕到更间接的证明，除非那条路更短更清楚。
- 新手详细模式优先多用中间步骤，不要把多个逻辑点压成一句 `simp`。

### 5. 每个等号链都要说明凭什么

- 常见“默认大家懂”的地方都要补：
  - 为什么 `E[w_t]=0`
  - 为什么交叉项为 0
  - 为什么 `E[\|e\|^2] = tr(P)`
  - 为什么 `tr(E[A]) = E(tr(A))`
  - 为什么有限求和能和期望交换
- 原则：如果读者第一次看到这条链，停下来问“这步凭什么”，那就补。

### 6. 抽象词要立刻落地到例子

- 像“投影到观测空间”这种话，如果不配例子就容易空。
- 立刻补一个当前文档里的具体例子：
  - `x=[p,v]^T`
  - `H=[1,0]`
  - 所以传感器只能直接看见位置误差，看不见速度误差

### 7. 汇总页不是公式墓碑

- KF 全流程汇总页不能只是：
  - `xhat`
  - `P`
  - `S`
  - `K`
- 应该把每个量翻译成人话，并在每个公式后补一句“这句到底在干嘛”。

### 8. 子 PDF / 独立推导的正确角色

- 用来放“逐步推导”“展开细节”“不适合塞进主线的大公式计算”。
- 不能把正文最关键的 Why / How 打包丢进去。
- 主 PDF 必须独立可读；子 PDF 只负责补细节，不负责补主线。

## 风格提醒

- 多写“为什么这一步自然会想到这个量”。
- 少写“这是著名的某某形式，所以很重要”。
- 优先说明量的朴素含义，再说它的数学定义。
- 如果一句解释让读者还会追问“什么意思”，说明这句还不够落地。
