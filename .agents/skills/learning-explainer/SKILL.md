# 学习型解释器

## 何时使用

满足任一项就用：

- 用户在学概念，而不是只求结论；
- 用户说“没懂”“再详细一点”“从直觉推到公式”；
- 内容里有符号、矩阵、概率、递推、梯度、指标；
- 预计用户会追问“这个量从哪来”“这个数什么意思”“为什么是这个公式”。

TeX 文档场景：本 skill 管内容，`tex-document-creator` 管排版与编译。

## 工作流

1. 先做困惑点审计。
   见 `references/confusion-audit.md`。
2. 先给真实问题。
   说明它在解决什么，不先抛定义。
3. 再给出场对象。
   输入、状态、输出、损失、指标都先翻译成人话。
4. 再给最小贯穿例子。
   先有语义，再有数字。
5. 先讲直觉，再讲推导。
   推导只在读者已经知道公式要表达什么之后出现。
6. 最后给对比或反例。
   说明它是什么，也说明它不是什么。

## 硬约束

- 不要出现裸符号。
  向量要说分量语义，矩阵要说行列语义。
- 不要先给玩具数字再补意义。
  先定义编码规则，再写数字。
- 权重至少解释：
  符号表示作用方向，绝对值表示作用强度。
- 偏置至少解释：
  默认分数、门槛或基线。
- 概率至少解释：
  它是哪件事发生的概率。
- 导数至少解释：
  它测的是哪条影响路径。
- 遇到递推或链式法则：
  先讲一步，再推广到一般情形；
  先分直接路径，再分间接路径。
- 遇到易混词：
  必须显式对比。
  例如 hidden feature vs hidden state。

## 常用讲解骨架

默认顺序：

1. 极简背景
2. 出场对象
3. 最小例子
4. 朴素直觉
5. 数学推导
6. 对比/反例
7. 一句话总结

## 交付前自检

- 是否先讲意义，再讲公式？
- 所有数字是否都有来源？
- 所有符号是否都有语义？
- 推导是否先给一步，再给一般式？
- 是否提前回答最可能的追问？

如果其中两项答不上来，说明解释仍然过度压缩。
