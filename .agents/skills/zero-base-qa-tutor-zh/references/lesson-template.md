# 零基础问题驱动讲义模板

## 1. 交互式回答模板

适合聊天窗口短回复：

1. 先说这一节在解决什么。
2. 再列 3 到 5 个“新手最该问的问题”。
3. 再逐个回答。
4. 回答顺序固定为：
   - 背景问题
   - 朴素直觉
   - 数学表达
   - 代码对应
5. 最后收束成一句“这一节真正该记住什么”。

## 2. 常用提问脚手架

### 背景问题

- 这个模块到底在解决什么现实困难？
- 如果没有它，旧方法会卡在哪里？
- 当前这一节在整章主线里处于哪一步？

### 朴素直觉

- 先别写公式，它到底在做什么？
- 为什么一个位置需要看别的位置？
- 为什么不能只用自己当前的表示？

### 数学表达

- 公式里每个量分别是什么？
- 这个分数/概率/权重到底在测什么？
- 为什么这里自然会想到点积、softmax、加权求和？

### 代码对应

- notebook 里哪一行在做这一步？
- 当前变量是上一层哪个数学对象？
- 代码里哪些数字是演示用，哪些是模型结构的一部分？

## 3. TeX 讲义模板

适合沉淀成文档时使用：

1. 这一节到底在解决什么
2. 新的讲解原则
3. 新手最该问的 N 个问题
4. QA 讲解
5. 把例子完整走一遍
6. 这一节和真实模型还差什么
7. 代码里每一步对应什么
8. 这一节真正该记住什么

如果要直接开始写 `.tex`，优先复用 `../assets/zero-base-lesson-template.tex`。

注意：

- 模板是默认脚手架，不是固定答案。
- 节的顺序、数量、命名都可以按用户任务调整。
- 如果用户要“更像教材”，可以加强 `Definition / Theorem / Proof / Example`。
- 如果用户要“更像讲义”或“更像科普”，可以弱化定理环境，增强叙述和图示。
- 如果用户已经给了自己的段落或比喻，先把用户原话吸收进模板，再考虑扩写。

## 4. 关键提醒

- “新的讲解原则”建议固定写成四条：
  - 背景问题
  - 朴素直觉
  - 数学表达
  - 代码对应
- 如果用户之前明确表示“不要截图驱动”，文档里要显式写出这套原则。
- 若文中出现 toy 版本与真实模型，必须专门留一节对比。
- 若文中出现具体数值，优先从 notebook 或代码单元核对，不要手写猜数。
